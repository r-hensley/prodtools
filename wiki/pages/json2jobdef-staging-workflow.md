# json2jobdef staging-config workflow

`data/<campaign>/*.json` is the per-stage entry catalog. `json2jobdef`
consumes one entry from one of those files and emits a `cnf.*.tar`
jobdef tarball ready for `submit_map` / `mu2ejobsub`. This page is the
**cross-stage map**: what each stage's JSON looks like, how dsconf
flows through the chain, and the standard add-an-entry-and-push loop.

For the CLI flag reference see `EXAMPLES.md` (regenerated by
`/refresh-examples`). For one stage's deep-dive see
[[pbi-sequence-workflow]]. For per-gotcha pointers see the **Known
gotchas** section below.

## The staging-config model

Each `data/<campaign>/<stage>.json` is a JSON array. Each element is
an entry describing one production output. `json2jobdef` selects
entries via `--index N` (positional), `--desc D --dsconf C` (named),
or `--dsconf C` alone (multi-select), then emits one or more
`cnf.mu2e.<desc>.<dsconf>.<idx>.tar` jobdefs.

Entries come in **two shapes**:

- **Scalar shape** (g4bl, stage1, primary_muon, pbi_sequence, digi,
  merge_filter, resampler_beam, resampler_stm) — each top-level field
  is a string/number, one entry produces one tarball.
- **Array shape** (mix, reco, evntuple) — most top-level fields are
  arrays of equal length. One entry produces one tarball per index in
  the array (cross-product of `dsconf[i]` × `fcl[i]` × `input_data[i]`
  × `fcl_overrides[i]` × …). This is the bulk-output form.

Array-shape entries do **not** carry a top-level `desc` — descriptor
is derived from `input_data` filenames. Select them by `--index` or
`--dsconf`.

| resampler-beam | `resampler_beam.json` | `pileup/{EleBeam,MuBeam,MuStopPileup,Neutrals,PiBeam}*.fcl`, `primary/RPC{External,Internal}Physical.fcl`, `beam/PhysicalPionStops.fcl`, `primary/TargetPiStopPreFilter.fcl`, `common/artcat.fcl` | dts/sim | dts (flash, etc.) | scalar | `resampler_name`, `sequential_aux` |
| resampler-stm | `resampler_stm.json` | `pileup/STM/{BeamTo2VD,BeamTo2VD1809,NeutralsTo2VD}.fcl` | dts | dts | scalar | `resampler_name`, `sequential_aux` |
| evntuple | `evntuple.json` | `EventNtuple/fcl/from_mcs-{mockdata,extracted}.fcl` | mcs | root | array | `dsconf` is `MDC2025-NNN` (campaign code, no `_best_vN`) |

**Tier names** map to the dot-delimited dataset prefix:
`<tier>.<owner>.<desc>.<dsconf>.<seq>.<format>` — `sim` for low-level
beam, `dts` for primary truth, `dig` for digitized, `mcs` for
reconstructed, `nts`/`root` for analysis ntuples, `etc`/`bck` for
ancillary.

**g4bl is decoupled from the Offline chain.** Despite living in the
same `data/<campaign>/` tree, `g4bl.json` entries drive a Geant4
Beamline runner natively on AL9 (`spack load g4beamline`, not
`mu2e -c`; no SL7/apptainer wrap since commit `401e3da` — see
[[g4bl-runner]] for the before/after detail) and emit `.root`
output. No other entry in `data/<campaign>/<stage>.json` references
g4bl outputs as input, so within prodtools g4bl outputs are
terminal. Treat g4bl as a parallel pipeline that prodtools also
schedules, not as stage 0 of the Offline chain. (Whether external
tools consume g4bl outputs is outside this repo's visibility.)

## Entry schema (superset)

Fields seen across the 12 files, grouped by role. **R** = required
where it appears; **O** = optional.

- **Identity**: `desc` (R for scalar entries; absent in array
  entries), `dsconf` (R), `tarball_append` (O), `owner` (O).
- **Pipeline**: `fcl` (R), `fcl_overrides` (O — list-of-dicts),
  `simjob_setup` (R — list with one cvmfs `setup.sh`).
- **Inputs**: `input_data` (R for stages consuming art), `inloc` (R
  for those — `tape`/`disk`/`dir:<path>`/`none`/`resilient`),
  `pileup_datasets` (mix only), `resampler_name` (resamplers and a
  few primary entries), `embed_dir` / `main_input` (g4bl only).
- **Job shape**: `events`, `njobs`, `events_per_job`, `run`, `outloc`,
  `outputs`.
- **PBI / mix specific**: `event_id_per_index`, `chunk_lines` (inside
  `input_data`), `pbeam`, `mixconf`, `merge_events`.
- **Flags**: `sequential_aux`.

### Cross-product expansion (array shape)

mix / reco / evntuple entries declare arrays of equal length on most
keys. One entry expands to one tarball per index; positional values
align across keys. Example — a reco entry like:

```json
{
  "dsconf":        ["MDC2025af_best_v1_1"],
  "tarball_append": "-reco",
  "fcl":           ["Production/JobConfig/recoMC/OnSpill.fcl"],
  "input_data":    [[{"dig.mu2e.CeEndpointMix1BBTriggered.MDC2025af_best_v1_1.art": 1}, ...]],
  "fcl_overrides": [[{"services.DbService.purpose": "Sim_best", "services.DbService.version": "v1_1"}]],
  "inloc":         ["tape"],
  "outloc":        ["tape"],
  "simjob_setup":  ["/cvmfs/.../Musings/SimJob/MDC2025af/setup.sh"]
}
```

… produces one cnf per array index (here, length 1). Adding more
entries to `dsconf`/`fcl`/`input_data`/etc. (all the same length)
emits more cnfs from one `json2jobdef` invocation.

## dsconf flow

| Hop | Input dsconf | Output dsconf | Note |
|---|---|---|---|
| stage1 → digi | `MDC2025ac` (etc.) | `MDC2025a{d,e,f,g,i}_best_v1_3` | digi adds the `_best_v<N>` calib-version suffix; the leading letter often bumps too. |
| digi → mix / reco | `MDC2025a{d,e,f,g,i}_best_v1_X` | preserved | reco's cnf basename is disambiguated via `tarball_append: "-reco"` so it doesn't collide with the digi cnf. |
| reco → evntuple | `MDC2025a{f,g,i}_best_v1_X` | `MDC2025-NNN` | evntuple diverges to a campaign-code dsconf (e.g. `MDC2025-002`), driven by an Analysis SimJob setup. See memory `reference_unmixed_chain_and_dsconf.md`. |
| resampler outputs | input dsconf | preserved | flash/STM resamplers keep their input's dsconf; outputs become pileup inputs. |

The `<simjob>_best_v<N>` convention: `<simjob>` matches a
`/cvmfs/.../Musings/SimJob/<simjob>/setup.sh`; `best` and the version
come from the Proditions DB purpose + version used at reco time.

## SimJob version selection

**Rule:** New entries default `simjob_setup` (and the dsconf prefix)
to the lexicographically-greatest existing release under
`/cvmfs/mu2e.opensciencegrid.org/Musings/SimJob/<campaign>xx/`.
Forward-only — don't rewrite existing entries silently.

```
ls /cvmfs/mu2e.opensciencegrid.org/Musings/SimJob/ | grep "^MDC2025" | sort | tail -1
```

returns the candidate. As of 2026-05-05 the MDC2025 series exposes
`aa…ak, am` (no `al`), with `am` the latest.

**Cross-check against known-broken releases before pinning:**

- `MDC2025aj` — `mu2e-trig-config/trigCalFilters.fcl` schema drift
  blocks any `mu2e -c` invocation that includes mixing. Memory
  `reference_mdc2025aj_trig_config_drift.md`.
- Anything older than `MDC2025aj` lacks the PBI event/subrun offset
  PRs — fine for non-PBI work, blocking for PBI sequence inputs.
  Memory `reference_pbi_event_offset_prs.md`.

If the latest release is broken for the stage you're configuring,
step back to the next-latest working one and note the reason inline
(`desc` suffix, sibling memory, or a comment alongside the entry).

The dsconf prefix should bump in lockstep with `simjob_setup`. A new
entry pointing at `…/Musings/SimJob/MDC2025am/setup.sh` should write
to `MDC2025am_best_v<N>_<M>`, not `MDC2025af_best_…`. Mixing the two
breaks the convention that the prefix identifies which Musings tree
produced the output.

## services.DbService rule

Digi, mix, and reco entries that consume calibrations need explicit
`fcl_overrides`:

```json
"services.DbService.purpose": "Sim_best",
"services.DbService.version": "v1_3"
```

`purpose = Sim_best`; `version` = the `vN_M` suffix of the entry's
`<simjob>_best_v<N>_<M>` dsconf. Without these, `mu2e -c` aborts at
`ProtonBunchTimeFromStrawDigis` with calibration set `EMPTY -1/-1/-1`.

**Coverage today** in `data/mdc2025/reco.json`: only the `ai_best_v1_3`
PBI reco entry sets the overrides; the older `af_best_v1_3` and
`ag_best_v1_3` entries do not and would fail a local smoke test
unchanged. Set the overrides on every new reco entry. Full incident
in memory `reference_reco_dbservice_overrides.md` and
[[pbi-sequence-workflow]] §"Stage 3 reco — DbService gotcha".

## Standard loop

```
1. Edit data/<campaign>/<stage>.json — add or update one entry
2. /mu2e-run json2jobdef --json data/<campaign>/<stage>.json --desc <D> --dsconf <C>
   → produces cnf.mu2e.<desc>.<dsconf>.<idx>.tar in cwd
3. Local smoke:
     jobfcl --jobdef cnf.*.tar --index 0 > test.fcl
     mu2e -c test.fcl
4. Production push:
     /mu2epro-run json2jobdef ... --prod
   then submit via submit_map / `/poms-push`
```

`/stage-entry <stage> [selectors]` is a thin wrapper over step 2 that
validates the stage name and forwards the selectors to `json2jobdef`.

### CLI flags (verified against `utils/json2jobdef.py:506-525`)

`--json` (R), `--desc`, `--dsconf`, `--index`, `--pushout`, `--prod`,
`--verbose`, `--no-cleanup`, `--jobdefs`, `--json-output`, `--extend`,
`--event-count-positive`, `--ignore-empty`. No `--setup` / `--code` —
the simjob setup comes from each entry's `simjob_setup` field.

`--prod` ⇒ `--pushout` + post-run `mkidxdef --prod` (creates the
`iMDC2025-NNN` SAM index definition).

### Selector semantics (the `--index` gotcha)

`--index N` is **N in the flattened (entry × input_data) expansion**,
not the JSON array position of the entry. With ten entries where the
first holds 11 inputs, `--index 7` selects the 8th input of entry 0,
not entry 7. Generic-tarball entries (no `input_data`) still
contribute one slot to the flat space.

**Recommended selectors:**

- **Bulk by output dsconf** (use this for array-shape stages like
  mix/reco/evntuple): `--dsconf MDC2025am_best_v1_3` expands every
  matching entry into one cnf per `input_data` element and registers
  them all in `jobdefs_list.json` in one call. Verified on reco
  2026-05-05 producing 19 cnfs across 3 entries (17 OnSpill + 1
  Extracted + 1 OffSpill).
- **By descriptor** (scalar entries with a `desc` field):
  `--desc <D> --dsconf <C>` picks one entry unambiguously.
- **By flat input position** (`--index N`): only useful when you
  already know the flattened ordering. Easy to miscount; prefer the
  other two.
